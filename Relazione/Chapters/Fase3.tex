\chapter{Phase 3: Hardening \& Mitigation}
This chapter presents the Blue Team response to the vulnerabilities identified and exploited in the previous phases. The objective is to implement a set of countermeasures that enforce the three fundamental security properties: \textbf{integrity}, \textbf{confidentiality}, and \textbf{authenticity}.

\medskip
\noindent Unlike Phase 2, where the malicious firmware was developed using the Mongoose OS framework to match the original Shelly ecosystem, the hardened firmware for Phase 3 was built entirely using the \textbf{Arduino IDE} (with the ESP8266 Arduino Core). This deliberate choice was made to vary the development approach and to demonstrate that the same hardware can be programmed using different toolchains. The Arduino ecosystem also provides direct access to the \textbf{BearSSL} cryptographic library, which is essential for implementing both \textit{MQTTS communication} and \textit{firmware signature verification} within the strict memory constraints of the ESP8266.

\medskip
\noindent The hardened firmware preserves the same functional capabilities as the Phase 2 variant: relay control, energy monitoring via the HLW8012, MQTT telemetry publishing, and an embedded web interface (Figure~\ref{fig:bluefw_ui}), but introduces two critical security layers:

\begin{enumerate}
    \item \textbf{Signed OTA Updates}: Enforce cryptographic signature verification to prevent the injection of unauthorized code.
    
    \item \textbf{MQTTS Migration}: Migrate from plaintext MQTT (1883) to MQTTS (8883) using TLS 1.2 with mutual X.509 certificate authentication.
\end{enumerate}

\section{Ensuring Integrity: Secure Update Implementation}
The most critical vulnerability exploited in Phase 2 was the \textbf{complete absence of cryptographic verification} during the OTA update process. In summary, this vulnerability was addressed by implementing a \textbf{digital signature scheme based on RSA-2048 with SHA-256 hashing}, utilizing the PKCS\#1 v1.5 padding scheme.

\medskip
\noindent The inclusion of the PKCS\#1 v1.5 padding scheme is a critical security requirement. Without padding (a vulnerability known as 'textbook RSA'), the signature scheme would be deterministic and mathematically homomorphic. An attacker could exploit the multiplicative property of raw RSA to forge valid signatures for arbitrary payloads (e.g., by multiplying known signatures together). 

\medskip
\noindent The approach follows a standard asymmetric cryptography model:

\begin{itemize}
    \item The firmware developer signs each binary using a private key.
    \item The device embeds the corresponding public key in its flash memory and uses it to verify the signature of any uploaded binary before applying the update.
\end{itemize}

\noindent The result is that any attempt to upload unsigned or tampered binaries is rejected.


\subsection{Firmware Signing}
The signing process is performed on the developer's machine using two Python scripts: \texttt{key.py} for RSA key pair generation, and \texttt{signature.py} for computing and appending the digital signature to the firmware binary.

\subsubsection{Key Pair Generation}
The first step is generating the RSA-2048 key pair. The script \texttt{key.py} (Figure~\ref{fig:key_gen}) uses the \texttt{cryptography} Python library to generate a 2048-bit RSA key pair with a public exponent of 65537 (a standard and secure choice).

\begin{figure}[H]
    \centering
    \includegraphics[width=0.80\linewidth]{Images/Fase3/GenerazioniChiaviFirma.png}
    \caption{RSA-2048 key pair generation script (\texttt{key.py})}
    \label{fig:key_gen}
\end{figure}

\noindent The key generation is executed with:

\begin{bashcode}
python key.py
\end{bashcode}

\noindent This produces two files:

\begin{itemize}
    \item \texttt{private\_key.pem}:, that is secret and it is used exclusively during the signing process.
    \item \texttt{public\_key.pem}, that is embedded directly into the firmware source code.
\end{itemize}


\subsubsection{Signing Process}
Once the firmware is compiled using the Arduino IDE (producing a \texttt{.bin} file), it is signed using the \texttt{signature.py} script (Figure~\ref{fig:firma_fw}). 

\medskip
\noindent The signing process is executed with the following command:

\begin{bashcode}
python signature.py ShellyPlugS.ino.bin
\end{bashcode}

\noindent The script operates according to the following steps:

\begin{enumerate}
    \item Read the entire firmware binary into memory.
    
    \item Compute the SHA-256 hash of the binary data.
    
    \item Sign the hash using the RSA-2048 private key with PKCS\#1 v1.5 padding, producing a 256-byte digital signature.
    
    \item Construct the signed binary by appending the 256-byte RSA signature and a 4-byte integer encoding the signature length to the original firmware data.
\end{enumerate}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.80\linewidth]{Images/Fase3/firmaFirmware.png}
    \caption{Firmware signing script (\texttt{signature.py}): computes the SHA-256 hash, signs it with RSA-2048 PKCS\#1 v1.5, and appends the signature to the binary.}
    \label{fig:firma_fw}
\end{figure}

\noindent Ultimately, this process yields a signed binary named \texttt{ShellyPlugS.ino\_firmato.bin} with the following internal structure:

\begin{itemize}
    \item \textbf{Original firmware binary}: $N$ bytes.
    \item \textbf{Signature}: $256$ bytes containing the RSA signature.
    \item \textbf{Signature length}: $4$ bytes encoding the length, which allows the device to locate and extract the signature from the end of the uploaded file during verification.
\end{itemize}



\subsection{Signing Verification}
The verification process occurs \textbf{on the device} during the OTA upload. The hardened firmware embeds the RSA-2048 public key directly in flash memory and uses the \texttt{BearSSL} library to perform signature verification before applying any update.

\subsubsection{Public Key Embedding}
The public key is stored in the \texttt{public\_key\_pem[]} array using the \texttt{PROGMEM} attribute, which instructs the ESP8266 compiler to place the data in flash memory rather than SRAM, preserving the limited heap space available for runtime operations (Figure~\ref{fig:chiavi}).

\begin{figure}[H]
    \centering

    \caption{Public key for OTA verification and TLS certificates for MQTTS, all embedded in the firmware source code as \texttt{PROGMEM} constants}
    \label{fig:chiavi}
\end{figure}

\subsubsection{Verification Handler}
The OTA update endpoint (\texttt{/update}) has been completely redesigned compared to the original Shelly firmware. Instead of blindly accepting any uploaded binary, the hardened firmware implements a \textbf{three-phase upload handler} (Figure~\ref{fig:verifica_firma}) that performs cryptographic verification using BearSSL:

\begin{figure}[H]
    \centering
    \includegraphics[width=0.80\linewidth]{Images/Fase3/codiceVerificaFirmaUpdate.png}
    \caption{OTA signature verification handler}
    \label{fig:verifica_firma}
\end{figure}

\noindent The verification logic operates across three upload phases:

\begin{enumerate}
    \item \textbf{\texttt{UPLOAD\_FILE\_START}}: When the upload begins, the handler initializes the BearSSL verification process by preparing three core components:
    \begin{enumerate}
        \item It loads the embedded public key and sets up the SHA-256 engine to hash the incoming firmware.
        \item It links the public key to the hash engine to enable signature verification.
        \item It registers both the hash engine and the verifier with the ESP8266 \texttt{Update} library using the \texttt{Update.installSignature()} method.
    \end{enumerate}
    
    \item \textbf{\texttt{UPLOAD\_FILE\_WRITE}}: Each incoming data chunk is written to flash and simultaneously fed into the SHA-256 hash computation, where the \texttt{update} library internally handles the incremental hashing.
    
    \item \textbf{\texttt{UPLOAD\_FILE\_END}}: When the upload completes, the \texttt{Update.end(true)} call triggers the final verification. The library:
    \begin{enumerate}
        \item Reads the last 4 bytes of the uploaded data to determine the signature length.
        \item Extracts the RSA signature (256 bytes) from the end of the binary.
        \item Computes the final SHA-256 hash of the firmware data.
        \item Verifies the signature against the computed hash using the embedded public key.
    \end{enumerate}
    
    If verification succeeds, the update is applied and the device reboots. If it fails, the update is \textbf{rejected} and the existing firmware remains intact.
\end{enumerate}

\noindent This countermeasure directly invalidates both MITM attack variants demonstrated in Phase 2. Any unsigned binary will be rejected.

\section{Ensuring Confidentiality and Authenticity}
The second major vulnerability exploited in Phase 1 was the use of \textbf{unencrypted MQTT}, which allowed any network observer to intercept telemetry data (power, energy, temperature) and control commands (relay on/off) in plaintext. In summary, this vulnerability was addressed by migrating to MQTTS and implementing mutual TLS authentication to ensure both confidentiality and authenticity.


\subsection{Secure MQTT (MQTTS)}
The hardened firmware replaces the standard \texttt{WiFiClient} used in Phase 2 with a \\ \texttt{BearSSL::WiFiClientSecure} instance, which wraps all TCP communication in a TLS 1.2 encrypted tunnel. The MQTT client (\texttt{PubSubClient}) is then instantiated on top of this secure transport, transparently encrypting all MQTT traffic (Figure~\ref{fig:mqtts0}).

\begin{figure}[H]
    \centering
    \includegraphics[width=0.55\linewidth]{Images/Fase3/mqtts0.png}
    \caption{Secure client initialization using \texttt{BearSSL::WiFiClientSecure}, with the \texttt{PubSubClient} instance layered on top of it to handle encrypted MQTT traffic.}
    \label{fig:mqtts0}
\end{figure}

\noindent The MQTT connection is configured to use \textbf{port 8883} (the standard MQTTS port) instead of the plaintext port 1883:


\subsubsection{NTP Synchronization}
TLS certificate validation requires an accurate system clock to verify certificate validity periods (the \texttt{notBefore} and \texttt{notAfter} fields in X.509 certificates). Since the ESP8266 does not have a Real-Time Clock (RTC) and loses its time reference on every power cycle, the firmware implements NTP (Network Time Protocol) synchronization at startup (Figure~\ref{fig:mqtts1}).

\subsubsection{TLS Configuration}
The \texttt{setupTLS()} function (Figure~\ref{fig:mqtts1}) configures the BearSSL client for mutual TLS authentication by loading three cryptographic objects (Figure~\ref{fig:chiavi}):

\begin{enumerate}
    \item \textbf{CA Certificate}: The Certificate Authority certificate used to verify the broker's identity (set via \texttt{setTrustAnchors()}).
    \item \textbf{Client Certificate}: The device's X.509 certificate, presented to the broker during the TLS handshake (set via \texttt{setClientECCert()}).
    \item \textbf{Client Private Key}: The ECC private key corresponding to the client certificate, used to prove ownership (set via \texttt{setClientECCert()}).
\end{enumerate}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.80\linewidth]{Images/Fase3/mqtts1.png}
    \caption{NTP synchronization and mTLS configuration}
    \label{fig:mqtts1}
\end{figure}

\noindent The TLS buffer sizes are explicitly configured to accommodate the ESP8266's limited memory, with the instruction \texttt{wifiClientSecure.setBufferSizes(4096, 512);}. This sets a 4096-byte receive buffer and a 512-byte send buffer, representing a careful trade-off between TLS functionality and the approximately 50\,KB of heap available on the ESP8266.


\subsubsection{Mitigation Effectiveness}
This countermeasure directly addresses the \textbf{Information Disclosure} threat identified in Phase 1:

\begin{itemize}
    \item \textbf{Before (Phase~1/2)}: All MQTT traffic was transmitted in plaintext on port 1883. Any attacker on the local network could use Wireshark to intercept power, energy, temperature, and relay state data.
    
    \item \textbf{After (Phase 3)}: All MQTT traffic is encrypted using TLS 1.2 on port 8883. Even if an attacker captures the packets, the payload is encrypted and appears unreadable in Wireshark.
\end{itemize}


\subsection{Mutual Authentication (mTLS)}
While standard TLS ensures confidentiality by encrypting the communication channel, it only authenticates the \emph{server} (broker) to the \emph{client} (device). Mutual TLS (mTLS) extends this by also requiring the client to present a valid certificate to the server, ensuring \textbf{bidirectional authentication}. This prevents unauthorized devices from connecting to the MQTT broker and unauthorized brokers from impersonating the legitimate server.


\subsubsection{Certificate Hierarchy and Generation}
The mTLS infrastructure is built on a \textbf{private Certificate Authority (CA)} using Elliptic Curve Cryptography (ECC) with the \texttt{prime256v1} (P-256) curve. ECC was chosen over RSA for the TLS certificates because it provides equivalent security with significantly smaller key sizes, which is critical for the memory-constrained ESP8266.

\medskip
\noindent The PKI hierarchy consists of four entities, each with its own certificate directory on the server (Figure~\ref{fig:certificati}):

\begin{figure}[H]
    \centering
    \includegraphics[width=0.75\linewidth]{Images/Fase3/Certificati.png}
    \caption{Certificate directory structure: separate directories for CA, Broker, Client (Shelly device), and Subscriber, each containing its certificate (\texttt{.crt}), private key (\texttt{.key}), and CSR (\texttt{.csr})}
    \label{fig:certificati}
\end{figure}

\noindent The certificates were generated using the following \texttt{openssl} commands:


\begin{enumerate}
    \item \textbf{Certificate Authority (CA):}The root CA key and certificate are generated inside the \texttt{CA/} directory:

\begin{bashcode}
# Generate CA private key (ECC P-256)
openssl ecparam -name prime256v1 -genkey -noout -out CA/ca.key

# Generate self-signed CA certificate
openssl req -new -x509 -sha256 -key CA/ca.key -out CA/ca.crt -days 3650 -subj "/CN=MyHomeCA"
\end{bashcode}

    \item \textbf{Broker Certificate:} The broker's private key, signing request, and certificate are generated inside the \texttt{Broker/} directory. An extensions configuration file (\texttt{server.ext}) is created to specify the Subject Alternative Name (SAN), key usage, and basic constraints required for IP-based addressing during the TLS handshake:

\begin{bashcode}
# Generate Broker private key (ECC P-256)
openssl ecparam -name prime256v1 -genkey -noout -out Broker/server.key

# Generate Certificate Signing Request (CSR)
openssl req -new -sha256 -key Broker/server.key -out Broker/server.csr -subj "/CN=74.161.73.178"

# Create the extensions configuration file
echo "authorityKeyIdentifier=keyid,issuer" > server.ext
echo "basicConstraints=CA:FALSE" >> server.ext
echo "keyUsage = digitalSignature, nonRepudiation, keyEncipherment, dataEncipherment" >> server.ext
echo "subjectAltName = @alt_names" >> server.ext
echo "[alt_names]" >> server.ext
echo "DNS.1 = 74.161.73.178" >> server.ext
echo "IP.1 = 74.161.73.178" >> server.ext

# Sign the CSR with the CA using the extensions file
openssl x509 -req -in Broker/server.csr -CA CA/ca.crt -CAkey CA/ca.key -CAcreateserial -out Broker/server.crt -days 3650 -sha256 -extfile Broker/server.ext
\end{bashcode}

    \item \textbf{Client Certificate (Shelly Device):} The client's key pair and certificate, which identify the Shelly Plug S device to the broker, are generated inside the \texttt{Client/} directory:

\begin{bashcode}
# Generate Client private key (ECC P-256)
openssl ecparam -name prime256v1 -genkey -noout -out Client/client.key

# Generate Certificate Signing Request (CSR)
openssl req -new -sha256 -key Client/client.key -out Client/client.csr -subj "/CN=ShellyPlugS"

# Sign the CSR with the CA 
openssl x509 -req -in Client/client.csr -CA CA/ca.crt -CAkey CA/ca.key -CAcreateserial -out Client/client.crt -days 3650 -sha256
\end{bashcode}

    \item \textbf{Subscriber Certificate:} The subscriber's certificate (used for monitoring tools like \\ \texttt{mosquitto\_sub}) are generated inside the \texttt{Subscriber/} directory:

\begin{bashcode}
# Generate Subscriber private key (ECC P-256)
openssl ecparam -name prime256v1 -genkey -noout -out Subscriber/sub.key

# Generate Certificate Signing Request (CSR)
openssl req -new -sha256 -key Subscriber/sub.key -out Subscriber/sub.csr -subj "/CN=Subscriber01"

# Sign the CSR with the CA
openssl x509 -req -in sub.csr -CA CA/ca.crt -CAkey CA/ca.key -CAcreateserial -out Subscriber/sub.crt -days 3650 -sha256
\end{bashcode}

\end{enumerate}

\noindent All four entities share the same CA trust anchor (\texttt{ca.crt}), creating a closed trust domain where only devices possessing a CA-signed certificate can participate in the MQTT communication.


\subsubsection{Mosquitto Broker Configuration}
The Mosquitto MQTT broker is configured to enforce mTLS by requiring client certificate verification. The configuration directives are defined in the broker's configuration file located at \texttt{/etc/mosquitto/conf.d/default.conf}.

\medskip
\noindent To enable this setup, the generated certificates and private keys must be copied from the build directories to the standard Mosquitto certificate directory at \texttt{/etc/mosquitto/certs/}.

\medskip
\noindent Once the files are in place, the configuration block in \texttt{/etc/mosquitto/conf.d/default.conf} is defined as follows:

\begin{bashcode}
listener 8883
cafile /etc/mosquitto/certs/ca.crt
certfile /etc/mosquitto/certs/server.crt
keyfile /etc/mosquitto/certs/server.key
require_certificate true
use_identity_as_username true
\end{bashcode}

\noindent Key configuration parameters:
\begin{itemize}
    \item \texttt{listener 8883}: The broker listens on the standard MQTTS port.
    \item \texttt{cafile}: The CA certificate used to validate both client and broker certificates.
    \item \texttt{certfile} / \texttt{keyfile}: The broker's own TLS certificate and private key.
    \item \texttt{require\_certificate true}: \textbf{This is the mTLS enforcement directive} where the broker will reject any client that does not present a valid, CA-signed certificate.
\end{itemize}


\subsubsection{Subscribing with mTLS}
To verify the correct operation of the MQTTS channel, the \texttt{mosquitto\_sub} command-line client was used with the subscriber's certificate and key. The following command connects to the broker on port 8883, presents the subscriber's certificate for mutual authentication, and subscribes to all topics:

\begin{bashcode}
mosquitto_sub -h 74.161.73.178 -p 8883 -t "#" -v \
    --cafile ca.crt --cert sub.crt --key sub.key
\end{bashcode}
